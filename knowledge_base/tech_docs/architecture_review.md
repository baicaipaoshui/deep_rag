# 架构评审记录

## 评审信息

- 日期：2023年9月20日
- 评审人：技术VP陈刚、架构师李伟、运维负责人王磊
- 评审对象：电商平台核心系统架构

## 一、当前架构概述

系统采用微服务架构，共12个服务，部署在Kubernetes集群上。主要服务包括用户服务、商品服务、订单服务、支付服务、搜索服务、通知服务等。数据库使用MySQL 8.0主从架构，缓存使用Redis Cluster，消息队列使用RabbitMQ。

## 二、性能瓶颈分析

### 2.1 服务间调用链过长

订单创建流程涉及5个服务的同步调用：订单服务→商品服务（查价格）→库存服务（扣库存）→优惠券服务（核销）→支付服务（创建支付单）。整个链路的P99延迟达到1.5秒。

**优化建议：** 将库存预扣和优惠券核销改为异步处理。先创建待确认订单，返回给用户后异步完成库存和优惠券操作。如果异步操作失败，触发补偿机制取消订单。预计可将P99从1.5秒降至400ms。

### 2.2 数据库瓶颈

当前MySQL主库承载了所有写操作和部分强一致性读，高峰期QPS达到5000+，CPU使用率经常超过80%。

**优化建议：**
- 将报表查询和数据分析迁移到从库
- 对订单表按用户ID做分库分表（建议16个库×64个表）
- 引入TiDB作为分析型数据库，承载OLAP查询

### 2.3 缓存击穿问题

双11活动期间，部分热点商品的缓存同时过期，导致大量请求穿透到数据库，数据库CPU飙升至100%，影响了2分钟的服务可用性。

**优化建议：**
- 热点商品的缓存设置随机过期时间（基础TTL + 随机偏移）
- 使用互斥锁（分布式锁）保证同一时刻只有一个请求回源
- 对极热点数据使用本地缓存+异步刷新

## 三、架构优化建议

### 3.1 引入CQRS模式

将读写路径分离：
- 写路径：API → 写入MySQL → 发布事件
- 读路径：事件驱动更新读模型（Elasticsearch + Redis）→ 读取优化视图

这样可以独立优化读写性能，读操作可以使用针对查询优化的数据结构。

### 3.2 服务网格化

当前服务间的限流、熔断、重试逻辑分散在各个服务中，维护成本高且不一致。

建议引入Istio服务网格，将这些跨切面关注点下沉到基础设施层。好处是：
- 统一的流量管理策略
- 服务间mTLS通信
- 可观测性开箱即用

### 3.3 事件驱动架构升级

当前RabbitMQ仅用于异步任务处理。建议扩展事件驱动的使用范围：
- 订单状态变更发布领域事件
- 库存变更事件触发搜索索引更新
- 用户行为事件驱动推荐系统

将RabbitMQ升级为Kafka，支持事件溯源和流处理。

### 3.4 容灾与高可用

当前系统部署在单个可用区，存在单点故障风险。

建议：
- 数据库和缓存实现跨可用区部署
- Kubernetes集群扩展到双可用区
- 制定灾难恢复演练计划，RTO目标30分钟，RPO目标5分钟

## 四、优先级排序

| 优先级 | 优化项 | 预期收益 | 实施难度 | 建议时间 |
|--------|--------|----------|----------|----------|
| P0 | 数据库慢查询治理 | 接口响应时间降50% | 低 | 2周 |
| P0 | 热点缓存防击穿 | 避免大促故障 | 低 | 1周 |
| P1 | 订单链路异步化 | P99从1.5s降至400ms | 中 | 4周 |
| P1 | 读写分离与分库分表 | 数据库压力降50% | 高 | 8周 |
| P2 | CQRS模式引入 | 读性能提升5倍 | 高 | 12周 |
| P2 | 服务网格化 | 运维效率提升 | 高 | 8周 |
| P3 | 事件驱动升级 | 系统解耦 | 高 | 12周 |

## 五、评审结论

1. 短期（1个月内）聚焦数据库优化和缓存防护，这是投入产出比最高的方向
2. 中期（3个月内）完成订单链路异步化和读写分离
3. 长期（6个月+）推进架构模式升级（CQRS、服务网格、事件驱动）
4. 容灾方案作为持续建设项，尽快启动双可用区规划
