# 后端性能优化指南

## 一、API响应时间优化

### 1.1 慢查询治理

当前系统存在12个P95响应时间超过500ms的API接口，其中商品搜索接口的P99达到2.3秒。

优化建议：
- 对商品搜索接口，将模糊搜索从MySQL LIKE查询迁移到Elasticsearch，预计可将响应时间从800ms降至50ms
- 对订单列表接口，添加复合索引（user_id, created_at），避免全表扫描
- 对报表统计接口，引入预计算机制，将实时聚合改为定时任务+缓存读取

### 1.2 N+1查询问题

订单详情接口存在典型的N+1查询问题：先查订单列表，然后逐条查询订单项和商品信息。一个包含20条订单的请求实际产生了61次数据库查询。

建议使用批量查询替代循环查询，或使用DataLoader模式合并请求。预计可将数据库查询次数从61次降至3次，响应时间从450ms降至80ms。

### 1.3 异步处理

以下操作不需要在API请求中同步完成：
- 发送通知邮件和短信
- 写入操作日志
- 更新搜索索引
- 生成缩略图

建议使用消息队列（RabbitMQ或Redis Stream）将这些操作异步化。预计可将订单创建接口的响应时间从650ms降至200ms。

## 二、缓存策略

### 2.1 多级缓存架构

建议采用三级缓存架构：

- **L1 本地缓存（Caffeine）**：缓存热点数据，如商品分类、配置项。TTL=5分钟，容量=1000条。
- **L2 分布式缓存（Redis）**：缓存会话数据、商品详情、用户信息。TTL=30分钟。
- **L3 数据库**：兜底数据源。

### 2.2 缓存一致性

- 对于强一致性场景（如库存），使用缓存旁路模式（Cache-Aside）：写操作先更新数据库再删除缓存
- 对于最终一致性场景（如商品详情），使用定时刷新+消息驱动更新
- 避免缓存击穿：对热点key使用互斥锁或提前续期

### 2.3 Redis优化

当前Redis内存使用已达12GB，存在大量过期未清理的key。

建议：
- 对不同类型的数据设置合理的过期时间，避免永久key堆积
- 大value（>10KB）使用压缩存储
- 使用Redis Cluster替代单节点，支持水平扩展

## 三、并发与连接池

### 3.1 数据库连接池

当前数据库连接池配置为最大50个连接，但监控显示高峰期经常达到48-50个，偶尔出现连接等待超时。

建议：
- 将最大连接数调整为100，最小空闲连接数设为20
- 设置连接最大生命周期为30分钟，避免长连接占用
- 启用连接泄露检测，超过60秒未归还的连接自动回收

### 3.2 线程池优化

- HTTP请求处理线程池：核心线程=CPU核数×2，最大线程=CPU核数×4
- 异步任务线程池：核心线程=CPU核数，最大线程=CPU核数×2
- 设置合理的队列容量和拒绝策略，避免OOM

### 3.3 限流与降级

- 对公开API启用令牌桶限流：100 QPS/用户
- 对非核心功能（如推荐、评论）启用熔断降级
- 制定分级降级预案：P1=仅保留核心交易链路，P2=关闭推荐和搜索

## 四、JVM与GC优化

### 4.1 当前问题

监控显示每天有2-3次Full GC，每次暂停时间约800ms，影响用户体验。

### 4.2 优化建议

- 将堆内存从4GB调整为8GB，新生代与老年代比例设为1:2
- 从CMS切换到G1垃圾回收器，设置最大暂停时间目标为200ms
- 定期分析heap dump，排查内存泄漏嫌疑对象

## 五、可观测性

### 5.1 指标监控

建议完善以下监控指标：
- API响应时间分位数（P50/P90/P95/P99）
- 错误率和错误码分布
- 数据库连接池使用率
- JVM内存和GC统计
- Redis命中率和延迟

### 5.2 链路追踪

引入分布式链路追踪（如SkyWalking或Jaeger），支持：
- 跨服务调用链可视化
- 慢调用自动标记和告警
- 按trace ID快速定位问题
